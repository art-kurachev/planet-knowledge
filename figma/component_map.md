# Figma → Code Component Map

Связь между нодами в Figma (рабочий файл макетов) и React-компонентами в коде.
Агент использует этот файл при синхронизации вёрстки с макетом.

**Figma файл:** Planeta (`L83bKaobb9yHmcjDrPXT9c`)

---

## Источник истины

**Figma = источник визуала** (размеры, отступы, цвета, шрифты, тексты).  
**Код = источник логики** (props, хуки, обработчики, структура компонентов, бизнес-логика).

При синхронизации агент **обновляет только визуальные CSS-свойства** в styles-объектах.  
Агент **никогда не трогает**: JSX-структуру, props-интерфейсы, хуки, обработчики, импорты, логику рендеринга — если только пользователь явно не попросил.

## Защита ручных правок (@figma-lock)

Если значение в коде намеренно отличается от Figma — ставится метка-лок:

```tsx
padding: 12, // @figma-lock — ручная правка, не синхронизировать
```

или блочный лок:

```tsx
// @figma-lock-start — кастомная логика hover-состояния
const hoverStyle = isHovered ? { background: colors.primary.muted } : {};
// @figma-lock-end
```

### Правила для агента

1. Перед правкой строки — проверить есть ли `@figma-lock`
2. Если лок найден — НЕ менять, добавить в отчёт как «пропущено (залочено)»
3. Блочные локи — не трогать ничего внутри
4. Если Figma и код расходятся без лока — обновить по Figma, упомянуть в отчёте
5. Если расхождение структурное (новый элемент, удалённый блок) — остановиться, спросить

Снятие лока:
- «сними лок на [компонент]» — удалить все `@figma-lock` из файла
- «синхронизируй принудительно» — игнорировать все локи

## Триггерные команды

| Команда | Что делает агент |
|---|---|
| `синхронизируй` | Проверить ноды → найти расхождения → обновить код (с учётом локов) |
| `обнови вёрстку` | То же самое |
| `синхронизируй принудительно` | Игнорировать локи, полная перезапись визуала по Figma |
| `проверь [название]` | Синхронизация одного компонента |
| `что изменилось в фигме?` | Только отчёт — показать diff без правки кода |
| `сними лок на [компонент]` | Удалить @figma-lock из файла компонента |

## Алгоритм синхронизации

1. Прочитать этот файл из репозитория (не из кеша)
2. `figma_get_design_changes` — получить список изменённых нод
3. Для каждой ноды: `figma_execute` depth=15, извлечь размеры, отступы, gap, radius, цвета, шрифты
4. Прочитать текущий .tsx файл по пути из таблицы маппинга
5. Найти расхождения (пропускать залоченные строки/блоки)
6. Обновить незалоченные стили
7. `figma_capture_screenshot` — визуальная проверка
8. Отчёт: что обновлено, что пропущено, были ли структурные конфликты
