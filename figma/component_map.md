# Figma → Code Component Map

Связь между нодами в Figma и React-компонентами в коде.
Используется агентом для отслеживания изменений и автоматической синхронизации.

---

## Источник истины

**Figma = источник визуала** (размеры, отступы, цвета, шрифты, тексты).
**Код = источник логики** (props, хуки, обработчики, структура компонентов, бизнес-логика).

При синхронизации агент **обновляет только визуальные CSS-свойства** в styles-объектах.
Агент **никогда не трогает**: JSX-структуру, props-интерфейсы, хуки, обработчики, импорты, логику рендеринга — если только пользователь явно не попросил.

---

## Защита ручных правок (Code Locks)

Если пользователь или Cursor поправили значение в коде и это отличается от Figma — ставится метка-лок. Агент обязан уважать локи при синхронизации.

### Формат лока в коде

```tsx
padding: 12, // @figma-lock — ручная правка, не синхронизировать
```

или блочный лок:

```tsx
// @figma-lock-start — кастомная логика hover-состояния
const hoverStyle = isHovered ? { background: colors.primary.muted } : {};
// @figma-lock-end
```

### Правила для агента

1. **Перед правкой строки** — проверить есть ли `@figma-lock` в комментарии на этой строке
2. **Если лок найден** — НЕ менять значение, добавить в отчёт как «пропущено (залочено)»
3. **Блочные локи** (`@figma-lock-start` ... `@figma-lock-end`) — не трогать ничего внутри блока
4. **Если Figma и код расходятся, но лока нет** — обновить код по Figma, упомянуть в отчёте
5. **Если расхождение существенное** (новый элемент, удалённый блок, изменение структуры) — остановиться, спросить пользователя

### Снятие лока

Пользователь может сказать:
- «сними лок на SummaryCard» — агент удаляет все @figma-lock из файла
- «синхронизируй принудительно» — агент игнорирует все локи и перезаписывает по Figma

## Триггерные команды

| Команда | Что делает агент |
|---|---|
| `синхронизируй` | Проверить все ноды → найти расхождения → обновить код (с учётом локов) |
| `обнови вёрстку` | То же самое |
| `синхронизируй принудительно` | Игнорировать локи, полная перезапись визуала по Figma |
| `проверь [название]` | Синхронизация одного компонента |
| `что изменилось в фигме?` | Только отчёт — показать diff без правки кода |
| `сними лок на [компонент]` | Удалить @figma-lock из файла компонента |

## Алгоритм синхронизации

### Шаг 1 — Прочитать маппинг
Агент читает этот файл из репозитория (не из кеша).

### Шаг 2 — Проверить изменения в Figma
```
figma_get_design_changes
```
Получить список изменённых нод. Если пусто — проверить все ноды из маппинга (fallback).

### Шаг 3 — Для каждой изменённой ноды
```
figma_execute — считать полное дерево ноды (depth 5+)
```
Извлечь: размеры, отступы, gap, radius, цвета, шрифты, тексты, иерархию.

### Шаг 4 — Прочитать текущий код с диска пользователя
Найти соответствующий .tsx файл по таблице маппинга.
Прочитать через Filesystem MCP.

### Шаг 5 — Найти расхождения (с учётом локов)
Сравнить значения из Figma с кодом. Искать:
- Изменённые размеры (width, height)
- Изменённые отступы (padding, gap)
- Изменённые цвета (fills, strokes)
- Изменённые шрифты (fontSize, fontWeight)
- Новые/удалённые элементы
- Изменённые тексты

**Пропускать строки с `@figma-lock`**.
**Пропускать блоки между `@figma-lock-start` и `@figma-lock-end`**.

### Шаг 6 — Обновить код
Внести правки только в незалоченные строки стилей.
Сохранить на диск пользователя через Filesystem MCP.

### Шаг 7 — Визуальная проверка
```
figma_capture_screenshot — снять скриншот ноды
```

### Шаг 8 — Отчёт
- Какие компоненты обновлены
- Что именно изменилось
- Что пропущено из-за локов
- Были ли структурные конфликты (требуют решения пользователя)
