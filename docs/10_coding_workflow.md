# Coding Workflow — правила работы агента при вёрстке

## Главный принцип

Вёрстка 1 в 1 с Figma. Pixel-perfect. Никаких допущений и приближений.

---

## Процесс

### 1. Уточнение перед стартом

Перед началом работы агент **обязан** задать вопросы, если:
- Неясен scope — что именно верстать (весь экран, отдельная карточка, компонент)
- Не определено поведение (hover, active, disabled)
- Есть адаптив или только десктоп
- Непонятна структура данных (что приходит с API, что мок)
- Есть конфликт между макетом и существующими правилами

**Не начинать работу, пока не прояснены все неоднозначности.**

### 2. Чтение макета через Figma MCP

Перед написанием кода — полный обход дерева через `figma-console`:
- Считать все размеры, отступы, gap, radius, шрифты, цвета
- Не угадывать — только реальные значения из узлов
- Подробности в `docs/09_figma_to_code.md`

### 3. Цветовые токены — в отдельном файле

Все цвета выносить в отдельный файл токенов (например `tokens/colors.ts`).
Источник истины — `design-tokens/colors.json` и семантические токены из `figma/naming_rules.md`.

### 4. React-компоненты

- Каждый логический блок макета = отдельный React-компонент
- Компоненты переиспользуемые: данные через props, мок отдельно
- Иерархия компонентов повторяет иерархию фреймов в Figma
- Подробные технические правила — в `docs/09_figma_to_code.md`

### 5. Правила вёрстки из репозитория

При вёрстке агент **обязан** учитывать:
- `figma/naming_rules.md` — стили, токены, отступы (сетка ×8), шрифт Inter
- `docs/08_constraints.md` — технические и дизайн-ограничения
- `docs/02_design_principles.md` — принципы UI (информационная плотность, читаемость, контраст)
- `design-tokens/colors.json` — Style ID для Figma API

### 6. Перепроверка и доведение до идеала

После первой итерации агент **обязан**:
1. Визуально сравнить результат с макетом (скриншот из Figma vs рендер)
2. Проверить все размеры, отступы, цвета, шрифты
3. Если есть расхождения — **исправить и повторить проверку**
4. Итерировать до полного совпадения с макетом

**Расхождение в 1px, неверный цвет, неточный gap — это баг. Исправлять.**

### 7. Перенос правок из браузерного превью в исходники

Когда правки CSS сделаны в браузерном превью (DevTools, live-редактирование), агент **обязан** перенести их в исходники:

1. **Идентификация элемента** — использовать `elementPath`, `elementHTML`, `reactComponent.name` из описания правки
2. **Поиск в коде** — искать по имени React-компонента и содержимому (текст, классы, атрибуты)
3. **Где менять**:
   - Inline-стили (`style={...}`) — редактировать объект стилей в JSX или `Record<string, React.CSSProperties>`
   - CSS Modules — искать класс в `*.module.css` по `elementClasses` или `selector`
   - Tailwind — менять `className` (удалить/добавить утилиты)
4. **Применение** — заменить `oldValue` на `newValue` для указанного `property`, сохранить единицы и формат
5. **Не коммитить** сгенерированные файлы (`dist/`, `node_modules/`, `.vite/`) — только исходники

## Чеклист качества

- [ ] Все значения считаны из Figma, ничего не угадано
- [ ] Цвета вынесены в файл токенов, inline hex отсутствует
- [ ] Компоненты разбиты по логике макета
- [ ] Данные через props, мок отдельно
- [ ] Сетка отступов кратна 8
- [ ] Шрифт — только Inter (Regular, Medium, Semi Bold)
- [ ] Результат визуально идентичен макету
- [ ] Перепроверка выполнена, расхождения устранены
