# Constraints

## Технические ограничения

### Шрифт
- **Inter** — единственный допустимый шрифт
- Доступные начертания: Regular, Medium, Semi Bold

### Figma API

1. **Всегда использовать стили** — `await node.setFillStyleIdAsync(styleId)`, никогда не хардкодить hex-значения
2. **Async API** — использовать `figma.getNodeByIdAsync()`, не `figma.getNodeById()`
3. **После `arrange_component_set`** — ID вариантов меняются; всегда заново читать `set.children` перед применением стилей
4. **Иконки** — стиль ставить только на Vector внутри Instance, не на сам Instance-фрейм
5. **Стили иконок** — в табах и похожих компонентах иконки используют `Secondary/BlueGray` в обоих состояниях (active и inactive)
6. **Скриншоты** — не делать для пользователя (он видит макеты в Figma). Использовать только когда скриншот нужен агенту для собственного анализа

## Дизайн-ограничения

- Светлая тема, без тёмного режима
- Основной цвет — синий
- Белые блоки как контейнеры
- Контрастный, читаемый, структурный UI — без декоративности
- Рабочая среда 8+ часов/день — визуальный комфорт критичен

## Что нельзя ломать

- Логику статусов
- Последовательность процессов
- Контекст возврата (пользователь должен понимать, куда вернётся)
- Прямоту действий
- Однозначность интерфейса

## Контекст среды

- 6–15 вкладок одновременно
- Параллельные мессенджеры и звонки
- Работа кусками по 5–20 минут с частыми прерываниями
- Интерфейс должен быть устойчив к прерыванию и понятен без "вспоминания"

## Цена ошибки

- Ошибка = деньги, срыв сроков, конфликт с руководством
- Критические действия не маскировать
- Destructive-действия не ставить рядом с подтверждающими
- Подтверждение — осмысленное, не декоративное

## Бизнес-ограничения

> _Заполнить: сроки, бюджет, приоритеты_
