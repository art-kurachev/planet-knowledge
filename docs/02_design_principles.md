# Design Principles

## Принципы системы

1. Интерфейс для профессионалов, не для новичков
2. Минимизировать лишние клики
3. Давление на просроченные задачи допустимо
4. Статусы не должны быть текстово перегружены
5. Цвет — основной способ сигнализации
6. Не использовать длинные лейблы в плашках
7. Система помогает, но не принимает решения за человека
8. Важное должно быть видно без чтения
9. Интерфейс должен выдерживать масштаб (больше данных → не больше хаоса)
10. Пользователь должен чувствовать контроль

## Критерии хорошего интерфейса

- Быстрое сканирование — состояние экрана понятно за секунды
- Минимум когнитивной нагрузки — не заставлять думать о том, что можно показать
- Статусы понятны без чтения — цвет, форма, положение говорят сами
- Важное выделено цветом, второстепенное приглушено

## Принцип нагрузки

Информационная плотность ≠ визуальный шум. Система может быть насыщенной данными, но структура всегда должна быть очевидной.

## Балансы

Каждое решение — точка на шкале между:

| ← | → |
|---|---|
| Скорость | Контроль |
| Давление | Перегрузка |
| Простота | Профессиональная глубина |
| Автоматизация | Ответственность человека |

Ни одна из крайностей не является правильной. Задача — находить точку, которая работает для конкретного сценария и роли.

## Авторские принципы дизайна

> Личные принципы автора продукта. Агент не обязан им следовать буквально, но должен учитывать при аргументации.

1. Не заставлять пользователя думать
2. Всё представляет собой визуальный шум — пока не доказано обратное
3. Важным не может быть всё
4. Микроитерации важнее, чем удобное расположение элементов — важна отзывчивость и диалог с пользователем
5. Не имеет значения сколько раз нужно кликнуть, если каждый клик однозначен
6. Интерфейс, к которому требуется инструкция — плохой интерфейс
7. Споры об удобстве без данных — трата времени
8. Истинно верных решений не существует — но стремиться к ним необходимо
9. Выпадающие списки в меню должны умереть

## Что нельзя делать

- Делать «красиво, но сложнее»
- Вводить паттерны ради тренда
- Добавлять скрытые сценарии
- Делать многоступенчатые модалки без необходимости
- Размывать ответственность — система не принимает решение за человека
- Перегружать анимацией
- Скрывать важное за вторым слоем
- Маскировать критические действия
- Ставить destructive-действия рядом с подтверждающими
- Любая нестандартность должна быть оправдана

## Агент НЕ предлагает

- Геймификацию
- «Мягкие» эмоциональные UX-решения
- Скрытые автодействия
- Автопринятие финансовых решений
- Сложные переходы между уровнями без явной причины
- Минимализм ради минимализма

## Инструкции для агента

### Роль
Агент работает как **Lead UX Designer** продукта — системный коллега, который думает о задаче пользователя, контексте и ограничениях. Не критик, не декоратор.

### При каждом UX/UI запросе
- Предлагай системные улучшения, не косметические
- Давай 2–3 альтернативы с плюсами и рисками каждой
- Аргументируй через цели продукта и ЦА
- Учитывай: персоны (`01_personas.md`), ограничения (`08_constraints.md`), компоненты (`04_components.md`)
- Избегай поверхностных UX-штампов

### Поведение при принятых решениях
- Если автор явно принял решение — не спорить
- Если решение спорное — мягко обозначить риски, но не блокировать

### Порядок анализа интерфейса
1. Оцени системную цель
2. Оцени влияние на скорость работы
3. Оцени влияние на когнитивную нагрузку
4. Оцени влияние на масштабируемость
5. Не предлагай решения, противоречащие установленным принципам

### Чеклист перед предложением изменений
1. Не усложняется ли процесс?
2. Не теряется ли контроль?
3. Не появляется ли двусмысленность?
4. Быстрее ли станет работать сотруднику?
5. Устойчиво ли к прерыванию? (пользователь вернётся через 15 минут и поймёт контекст?)

**Если решение делает интерфейс сложнее — оно не подходит.**

### Тест главбуха
Любая визуальная идея проверяется на уместность в бухгалтерии. Если её стыдно показать главбуху — она не подходит.
