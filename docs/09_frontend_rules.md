# Frontend Rules

Правила вёрстки фронтенда Planet'ы. Обязательны к соблюдению при создании и рефакторинге компонентов.

## Структура файлов

```
frontend/src/{module}/
├── components/          # UI-компоненты модуля
│   ├── icons/           # SVG-иконки как React-компоненты
│   │   └── index.ts     # Реэкспорт всех иконок
│   ├── shared/          # Переиспользуемые элементы (кнопки, бейджи и т.д.)
│   │   └── index.ts
│   ├── {feature}/       # Подкомпоненты сложного компонента
│   │   └── index.ts
│   └── {Component}.tsx
├── mocks/               # Моковые данные (отдельно от компонентов)
│   └── index.ts
├── tokens/              # Дизайн-токены (цвета, отступы)
│   └── colors.ts
├── {Module}.tsx          # Корневой компонент модуля
├── {module}.css          # Глобальные CSS-стили модуля (hover, медиа-запросы)
└── index.ts              # Публичный API модуля
```

## Правило 1: Не дублировать компоненты

Если элемент появляется в двух и более местах — он должен быть вынесен в `shared/` или `icons/`.

Примеры: кнопка PDF, иконка скачивания, иконка предупреждения, стрелки направления.

Если в процессе работы замечаешь копипасту — вынеси сразу, не откладывай.

## Правило 2: Моки отдельно от компонентов

Тестовые данные (MOCK_*) живут в папке `mocks/`, а не внутри компонентов.

Каждый мок-файл экспортирует и данные, и типы (интерфейсы). Компоненты импортируют типы и дефолтные данные оттуда.

Это нужно, чтобы при подключении реального API не лазить внутрь компонентов.

## Правило 3: Большой компонент — разбивай

Если компонент > 300 строк — его нужно разбить на подкомпоненты.

Подкомпоненты кладутся в одноимённую папку: `components/{feature}/`.

Каждый подкомпонент — самостоятельный файл со своими стилями.

## Правило 4: Иконки — отдельные файлы

Каждая SVG-иконка — это отдельный React-компонент в `components/icons/`.

Все иконки реэкспортируются через `icons/index.ts`.

Именование: `{Name}Icon.tsx` (например `DownloadIcon.tsx`, `WarningIcon.tsx`).

Если иконка имеет несколько размеров — суффикс размера: `WarningIcon16`, `WarningIcon12`.

## Правило 5: Цвета только из токенов

Все цвета берутся из `tokens/colors.ts`. Захардкоженные hex/rgb в компонентах запрещены.

Единственный источник правды — файл `design-tokens/colors.json` в корне репо. Фронтовый `tokens/colors.ts` должен ему соответствовать.

## Правило 6: Стили — inline objects

Стили пишутся как `const styles: Record<string, React.CSSProperties>` в конце файла компонента.

CSS-файл модуля (`{module}.css`) используется только для того, что нельзя сделать inline: hover-состояния, медиа-запросы, кастомные скроллбары.

Не смешивать подходы внутри одного компонента — если стиль в JS-объекте, не дублировать его в CSS.

## Правило 7: Шрифт

Единственный шрифт — **Inter**. Доступные начертания: 400 (Regular), 450, 500 (Medium), 600 (Semi Bold).

Всегда указывать `fontFamily: 'Inter, sans-serif'` на корневом элементе карточки/модуля.

## Правило 8: Реэкспорт через index.ts

Каждая папка (`icons/`, `shared/`, `mocks/`, `{feature}/`) имеет `index.ts` с реэкспортом.

Импорт из соседних папок — через `index.ts`, не через прямые пути к файлам.

## Правило 9: @figma-lock

Секции, которые привязаны к Figma-макетам и не должны перезаписываться автоматической синхронизацией, оборачиваются комментариями:

```tsx
{/* @figma-lock-start — описание */}
...
{/* @figma-lock-end */}
```

Стили залоченных секций тоже помечаются:

```ts
// @figma-lock-start — описание
chartSection: { ... },
chartWrap: { ... },
// @figma-lock-end
```

## Правило 10: Адаптивность

Минимальная поддерживаемая ширина — 768px (планшет).

Медиа-запросы для адаптива пишутся в CSS-файле модуля, не в inline-стилях.

Сетка — CSS Grid. Колонки схлопываются в 1 на 768px.
